# بوت شركة الشمعة للملكية الفكرية — دليل الرفع على الاستضافة

## 1. الملفات المطلوبة
تأكد إن كل هالملفات موجودة سوا بنفس المجلد على السيرفر:
- `bot.py`
- `faq_data.py`
- `requirements.txt`
- `files/` (مجلد فيه أي ملفات PDF/Excel بيرسلها البوت للمستخدم — راجع `faq_data.py`
  ودور على أي node فيه `"file": "اسم_الملف"` وتأكد إن هيدا الملف موجود فعلياً جوا `files/`
  بنفس الاسم بالضبط، وإلا البوت رح يرد "الملف غير موجود".)

## 2. تثبيت المتطلبات
```bash
pip install -r requirements.txt
```
(أو حسب واجهة "Setup Python App" على cPanel، فيها زر لتثبيت المتطلبات من نفس الملف.)

## 3. التوكن (BOT_TOKEN) — **الأهم أمنياً**
- **ما تحط التوكن مكتوب مباشرة جوا الكود أبداً.**
- انسخ `.env.example` وسمّيه `.env`، وحط فيه التوكن الحقيقي:
  ```
  BOT_TOKEN=هون_التوكن_الحقيقي
  ```
- إذا لوحة cPanel فيها خيار "Environment Variables" (بميزة Setup Python App)، الأفضل
  تحط `BOT_TOKEN` فيها مباشرة بدل ملف `.env` — أكثر أماناً.
- ملف `.env` ما لازم يكون قابل للوصول من المتصفح إطلاقاً (لازم يكون خارج مجلد
  `public_html` أو محمي بصلاحيات وصول). إذا حدا قدر يفتحه من الإنترنت، تصير عندك
  مشكلة أمنية خطيرة (أي حدا بقدر ياخد التوكن ويتحكم بالبوت بالكامل).

## 4. طريقة التشغيل — في نقطة مهمة لازم تتأكد منها
البوت حالياً شغال بنظام **Polling** (`app.run_polling()`) — يعني لازم يضل
شغال كعملية دائمة (Long-running process) على مدار الوقت، مش زي موقع عادي
بيرد بس لما حدا يزوره.

**اسأل مدير الاستضافة (الشخص يلي عنده وصول cPanel) هالسؤال بالتحديد:**
> "هل عندك بلوحة cPanel خيار Setup Python App / Python Selector، وفيه إمكانية
> إبقاء تطبيق بايثون شغال بشكل دائم بالخلفية؟"

- **إذا الجواب نعم** → بيقدر يشغل `bot.py` مباشرة من خلال "Setup Python App"
  (بيحدد `bot.py` كملف بدء التشغيل)، وهيك البوت رح يضل شغال بشكل طبيعي.
- **إذا الاستضافة أساسية (مواقع HTML بس، بدون Python App Manager)** → غالباً
  ما رح تقدر تشغل هالبوت عليها إطلاقاً بالطريقة الحالية (Polling)، لأنو
  الاستضافات المشتركة العادية بتوقف أي عملية طويلة الأمد تلقائياً. بهاد
  الحالة، الحل هو تحويل البوت لنظام **Webhook** بدل Polling (يحتاج تعديل
  إضافي بالكود + شهادة SSL على الدومين) — إذا وصلنا لهاد السيناريو، ارجع
  اسألني وبعدلّك الكود المناسب.

## 5. تشغيله بشكل دائم (لو في وصول SSH)
لو الشخص عنده وصول SSH (مو بس cPanel)، الأفضل يشغل البوت من خلال أداة
تخليه يشتغل تلقائياً ويعيد تشغيله لو وقع، متل:
```bash
# مثال باستخدام screen (بسيط)
screen -S chamaa_bot
python3 bot.py
# اضغط Ctrl+A ثم D للخروج بدون ما توقف العملية
```
أو الأفضل استخدام `systemd` أو `supervisor` لو متوفرين على السيرفر.

## 6. تدقيق أمني قبل الرفع (Checklist)
- [ ] التوكن مو مكتوب بالكود، وموجود بس كمتغير بيئة أو بملف `.env` غير مرفوع لـ GitHub
- [ ] ملف `.gitignore` موجود وفيه `.env` (مرفق هون جاهز)
- [ ] الريبو على GitHub **خاص (Private)** مو عام، خصوصاً إذا فيه أي معلومة حساسة
- [ ] مجلد `files/` ما فيه غير الملفات يلي فعلاً بدك تشاركها مع المستخدمين
      (تأكد ما حطيت فيه بالغلط ملفات فيها معلومات داخلية/عقود/بيانات عملاء)
- [ ] صلاحيات ملف `.env` على السيرفر محدودة (600) بحيث بس المستخدم المسؤول
      يقدر يقرأه
- [ ] تحديث المكتبات بشكل دوري (`pip install -U -r requirements.txt`) لسد أي
      ثغرات معروفة

## 7. ملاحظة على التعديل الأمني اللي تم بالكود
تم تعديل جزء إرسال الملفات (`file:`) بحيث البوت **ما يقدر يرسل غير ملفات
موجودة فعلياً جوا مجلد `files/`** — هيك حتى لو حدا حاول يتلاعب بالطلب
(callback data) ليطلب ملف من مكان تاني على السيرفر (مثل ملفات إعدادات أو
كود المشروع نفسه)، البوت رح يرفض ويقول "الملف غير موجود" بدل ما يكشف أي
ملف حساس.
